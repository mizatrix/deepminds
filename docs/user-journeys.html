User journeys

A map of how each role moves through the CS Excellence portal — from arrival to outcome — and where the system enforces access.

Live URL: csexcellence.msa.edu.eg Auth: NextAuth v5 + Google OAuth Domain restriction: @msa.edu.eg

Overview

The portal is a public showcase with a gated submission and review workflow. Anyone can browse the home page, leaderboard, and approved achievements. Submitting an achievement, viewing personal data, or reviewing submissions requires an authenticated MSA Google account. Roles fan out from there.

Roles

Student

Super Admin

Everything Admin can do, plus role grants and system settings.

2. Sign-in flow

One button: Continue with Google. The server enforces the MSA domain after Google returns — a personal Gmail surfaces a friendly error naming the rejected address, not a generic failure.

sequenceDiagram
    autonumber
    participant U as User
    participant P as Sign-in page
    participant G as Google OAuth
    participant CB as signIn callback
    participant DB as Database
    participant JWT as Session JWT

    U->>P: Click "Continue with Google"
    P->>G: Redirect with hd=msa.edu.eg
    G-->>U: Account chooser
    U->>G: Select @msa.edu.eg account
    G->>CB: Profile and tokens
    CB->>CB: Check email ends with @msa.edu.eg
    alt domain mismatch
        CB->>P: return false + rejected-email cookie
        P-->>U: Show "isn't an MSA account" error
    else domain OK
        CB->>DB: findUserByEmail
        alt new user
            CB->>DB: createUser role=STUDENT
        else deactivated
            CB->>P: return false
        end
        CB->>JWT: attach id and role from DB
        JWT-->>U: Session cookie set
        U->>P: Redirect to callbackUrl
    end
Role-into-JWT is critical: the signIn callback attaches dbUser.role to the NextAuth user object so the jwt callback can persist it. Without this step the JWT defaults to STUDENT and proxy.ts redirects every admin away from /admin/*. Changing a user's role in the DB requires that user to sign out and back in once to refresh the JWT.

What the student sees after review

On approval: in-app notification, optional points and badges, the submission appears on the public leaderboard if it qualifies, and a certificate may be issued. On rejection: notification with the admin's feedback so the student can revise and resubmit.

Things worth knowing

Students see every submission they have made on /student/dashboard — sorted by most recent first — together with aggregated stats (total / approved / pending) and total points.

  • No pagination yet. Fine up to a few hundred entries. Easy follow-up if a student ever crosses ~50 submissions.
  • Edit while pending. A student can revise a PENDING submission via the pencil icon next to its row — backed by updateStudentSubmission. Once an admin approves or rejects it, only admins can change it.
  • Public portfolio. /student/portfolio renders the shareable, public-facing view of a student's approved submissions. The link works for visitors who are not signed in.
  • No server-side drafts. A submission is either posted (status=PENDING) or held in the browser's sessionStorage until the user returns. There is no drafts collection in the database today.

Access gates

Five points in the codebase enforce who can do what. They run in this order on every gated request.

Gate Location Effect
Authentication required proxy.ts No session → redirect to /sign-in?callbackUrl=…
Admin route role check proxy.ts STUDENT on /admin/* → redirect to /student/dashboard
Domain whitelist lib/auth/config.ts · signIn Email must end with @msa.edu.eg or sign-in returns false
Deactivated account lib/auth/config.ts · signIn isActive=false → sign-in rejected
Role hand-off into JWT lib/auth/config.ts · signIn → jwt Attaches dbUser.role to NextAuth user so the JWT carries it

Open notes

Submission form lives on the public home page

Anonymous visitors can start filling in a submission and only hit the sign-in wall at upload time. The form preserves draft state via sessionStorage and returns the user after authentication. It works, but it invests effort before the wall is visible. A clearer pattern is to show a "Sign in to submit" CTA upfront for anonymous users.

Legacy /dashboard/* routes

The /dashboard, /profile, and /stats routes appear to be prototype leftovers that read from a hardcoded lib/data stub. Current navigation does not link to them. Candidate for deletion in a follow-up dead-code PR.

JWT staleness on role change

If a SUPER_ADMIN promotes a user from STUDENT to ADMIN, the target user must sign out and back in once before their JWT carries the new role. The default JWT lifetime is 30 days with 24-hour rolling updates. There is no server-side session invalidation today.
